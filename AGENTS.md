# 项目规则

## 项目背景

- 引擎：Cocos Creator 3.8.8。
- 语言：TypeScript。
- 目标平台：抖音小游戏。
- 游戏方向：挖地小子式挖矿循环 + Roguelite 随机增益。
- 原型资源策略：先使用占位图形，不让正式美术阻塞玩法验证。
- 玩法和经济设计必须优先符合抖音小游戏市场习惯：短局、强反馈、轻成长、低理解成本、竖屏单手体验、真机快速验证。

## 开发边界

- 玩法、UI、核心逻辑、配置代码都不能直接调用 `tt.*`。
- 所有平台能力必须经过 `assets/scripts/platform/`。
- Cocos 预览环境必须使用 Mock 平台实现。
- 抖音运行环境必须使用抖音平台实现，并提供错误降级。
- 默认不修改 `.scene`、`.prefab`、`.meta` 文件，除非任务明确要求做 Cocos 编辑器挂载。
- 不新增 npm 依赖，除非任务明确批准。
- 每次任务只改必要模块，避免顺手重构无关代码。

## 目录约定

```text
assets/
  scripts/
    core/
    gameplay/
    ui/
    platform/
    config/
```

- `core`：游戏状态、事件、存档、一局流程生命周期。
- `gameplay`：玩家、挖矿、矿块、地图生成、氧气、背包、增益。
- `ui`：首页、HUD、暂停、结算、升级、增益选择。
- `platform`：`PlatformManager`、`MockPlatform`、`DouyinPlatform`、平台接口。
- `config`：矿石、升级、增益、经济、深度数值配置。

## 平台规则

- 业务代码只依赖 `PlatformManager`，不依赖平台全局对象。
- `MockPlatform` 必须支持没有抖音 API 的 Cocos 预览环境。
- `DouyinPlatform` 负责登录、本地存储、分享、激励视频、震动、平台错误处理。
- 平台能力失败时必须优雅降级，不能阻塞核心挖矿循环。

## 文档规则

- 文档保持短、可执行、及时更新。
- 玩法方向、平台假设、提审约束、模块职责变化时，要同步更新相关文档。
- 优先使用表格和清单，不写长篇策划案。
- 修改代码结构、命名、模块边界或错误处理时，必须参考 `docs/CODING_STANDARDS.md`。
- 修改状态机、平台层、随机生成、增益系统、存档结构时，必须先查阅 `docs/ENGINEERING_LESSONS.md` 的对应检查清单。

## 工程防错规则

- 平台初始化只有成功才能缓存；失败必须允许后续重试。
- 事件顺序必须明确，事件 payload 要包含监听方所需数据，不能依赖监听方读取半更新的全局状态。
- Resolver 默认必须是纯函数，返回 delta；只有拥有状态的 Manager / GameState 才能修改状态。
- 所有增益效果必须经过统一 modifiers，不允许在消费端偷偷写特例。
- 随机概率叠加后必须保证总和不超过 1，必要时显式归一化。
- `RunState` / `SaveData` 克隆逻辑只能有一个共享入口。
- 公共方法不要用会隐式调用 `require...()` 的默认参数。
- Mock 可以降级返回 fallback data，但真实错误必须保持 `ok: false`。

## 每次任务完成后必须输出

1. 改了哪些文件。
2. 每个文件做了什么。
3. 需要在 Cocos 编辑器里手动挂哪些节点、组件或属性。
4. 如何在 Cocos 预览中测试；涉及平台能力时，也要说明如何在抖音开发者工具中测试。
5. 如果改动会影响手机真机表现、触摸输入、屏幕适配、资源、场景或构建配置，必须提醒：需要重新构建到抖音开发者工具后再手机验证。

## 验证规则

- 修改 `core/`、`gameplay/`、`skill/` 后必须运行 `npm.cmd test` 和 `npm.cmd run typecheck`。
- 只修改 UI 展示且不影响玩法状态时，至少运行 `npm.cmd run typecheck`。
- 修改平台、存档、结算、支付/广告入口时，必须检查失败结果是否能被 UI 日志或后续错误上报观察到。
