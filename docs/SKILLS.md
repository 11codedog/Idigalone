# 技能系统设计

## 设计前提

技能系统必须服务于抖音小游戏市场习惯：短局、强反馈、轻成长、低理解成本、竖屏单手体验。技能不是复杂天赋树，而是让玩家产生“这次我能挖得更爽一点”的明确感受。

第一版只做轻量框架：默认启用 1 个技能，提供技能展示页，不做复杂流派、不做多套装配、不做技能抽卡。

## 系统定位

| 系统 | 生效方式 | 作用 | 示例 |
|---|---|---|---|
| 装备升级 | 局外永久成长 | 稳定提升基础能力 | 氧气瓶提高氧气上限 |
| Roguelite 增益 | 每局三选一 | 本局制造变化 | 高价出货、富矿嗅觉 |
| 技能系统 | 解锁后携带生效 | 改变玩法规则 | 矿石压缩 |

边界原则：

- 装备升级解决“我变强了”。
- Roguelite 增益解决“这局不一样”。
- 技能系统解决“我换了一种挖法”。

## 第一版只做：矿石压缩

| 项目 | 设计 |
|---|---|
| 技能名 | 矿石压缩 |
| 类型 | 背包技能 |
| 效果 | 同类矿石最多 5 个压缩成 1 组，每组占 1 格背包 |
| 第一版状态 | 默认启用，不进存档，不做解锁 UI |
| 体验目标 | 明显延长单局探索时间，让玩家更愿意继续下挖 |

示例：

| 持有矿石 | 无压缩占用 | 有矿石压缩占用 |
|---|---:|---:|
| 铜矿 x1 | 1 | 1 |
| 铜矿 x5 | 5 | 1 |
| 铜矿 x6 | 6 | 2 |
| 铜矿 x13 | 13 | 3 |
| 铜矿 x5 + 银矿 x5 | 10 | 2 |

计算公式：

```text
单类矿石背包占用 = ceil(矿石数量 / 5) * 1
总背包占用 = sum(所有矿物背包占用)
节省格数 = 原始占用 - 压缩后占用
```

## 当前 UI 入口

| 入口 | 内容 | 边界 |
|---|---|---|
| 首页“技能”按钮 | 打开当前已启用技能页 | 只展示，不做技能解锁 |
| 技能页 | 显示技能名、启用状态、效果说明、占格示例 | 不修改存档，不影响局内输入 |
| 技能页“开始下矿” | 进入增益选择流程 | 与首页开始下矿一致 |

第一版技能页的目标是解决“功能存在但玩家看不见”的问题。后续如果要做技能解锁，再扩展 `SaveData`，并把技能页从“展示页”升级为“解锁/携带页”。

## 已落地代码结构

```text
assets/scripts/skill/
  SkillTypes.ts
  SkillConfig.ts
  SkillManager.ts
  InventoryCalculator.ts
```

| 模块 | 职责 |
|---|---|
| `SkillTypes` | 技能 ID、技能分类、技能状态和 modifiers 类型 |
| `SkillConfig` | 技能名称、描述、默认携带技能、压缩参数 |
| `SkillManager` | 提供当前启用技能和技能 modifiers |
| `InventoryCalculator` | 根据矿石数量和技能 modifiers 计算背包占用 |
| `MiningScreenView` | 页面路由，负责把 `skills` 状态分发到技能页 |
| `SkillsScreenView` | 展示当前启用技能、规则效果和背包占格示例 |
| `MiningDebugPanel` | 负责技能页路由切换，不计算技能效果 |

数据流：

```text
挖到矿石
  -> TileEffectResolver 返回 inventoryDelta
  -> RunManager 判断 delta 后背包是否还能容纳
  -> RunManager 更新 inventory
  -> InventoryCalculator 重新计算 backpackUsed
  -> HUD 显示 used/capacity
  -> 技能页/结算页展示矿石压缩效果
```

## 工程约束

- `TileEffectResolver` 只返回 delta，不计算背包规则。
- 背包占用只能由 `InventoryCalculator` 计算。
- `RunManager` 只负责流程和状态写入，不写具体技能特例。
- UI 只展示结果，不参与技能规则计算。
- 第一版不扩展 `SaveData`，后续做技能解锁时再加入存档结构。
